## IDEA 1 — vLLM Multi-Step Scheduling (CPU-overhead amortization)

### What it is

Amortize the 5–13 ms per-step CPU bubble (scheduler, sampler, detokenizer, host-device sync) by batching N decode steps before returning control to Python, eliminating N-1 out of every N scheduler round-trips.

### Why it might help here

The decode budget breakdown shows ~30 pp GPU-busy small-kernel tail and ~2.17 pp idle. A 5–13 ms bubble per step at M=8 MTP means that bubble is actually spread across 8 tokens — but the CPU-scheduling overhead is still paid per *draft-verify cycle*, not per individual token. At conc=1 single-stream, every CPU round-trip is fully exposed on the critical path. vLLM Multi-Step PR #7000 was merged into vLLM main; the question is whether the version deployed in this stack exposes the `--num-scheduler-steps` flag or equivalent and whether it was enabled in the current best checkpoint recipe.

This is NOT about changing emitted tokens — Multi-Step scheduling is transparent to the model; it just pipelines decode steps. The greedy-exact gate is unaffected.

### Expected magnitude

vLLM blog reports 1.4–2.1x improvements in throughput-oriented settings. For single-stream conc=1 the gain is smaller because there is no cross-request batching benefit, but the raw CPU overhead elimination still applies. A conservative estimate for our A10G single-stream setting: +2–6% TPS (+10–30 TPS). Could push us from 481 to 490–510.

### Key papers / references

- vLLM Multi-Step Scheduling RFC: https://github.com/vllm-project/vllm/issues/6854
- vLLM PR #7000 (MERGED): https://github.com/vllm-project/vllm/pull/7000
- vLLM v0.6.0 release blog: https://blog.vllm.ai/2024/09/05/perf-update.html

### Implementation guidance

**File to touch:** `target/serve.py` or whatever script wraps the vLLM `LLM` / `AsyncLLMEngine` constructor.

**Flag to add:**
```python
engine_args = AsyncEngineArgs(
    ...
    num_scheduler_steps=8,   # amortize over N decode steps
    # or, if using SamplingParams-level multi-step:
    multi_step_stream_outputs=False,  # avoid streaming overhead
)
```

**Critical check:** Confirm vLLM version in the docker image supports `--num-scheduler-steps`. Run `python -c "from vllm import LLM; help(LLM)"` and look for `num_scheduler_steps`. If on vllm>=0.6.0 it should be present.

**Gotcha:** Multi-step scheduling interacts with speculative decoding. If MTP/spec-dec is already handling its own step bundling, adding `num_scheduler_steps>1` may be a no-op or cause double-counting. Profile with `--num-scheduler-steps=1` (baseline) vs `=4` vs `=8` to find the sweet spot.

**Diagnostic first:** Add `VLLM_TRACE_FUNCTION=1` and check whether Python-side scheduling time shows up in nsys trace. If the GPU-trace shows idle gaps between kernel clusters matching the scheduler period, Multi-Step will help.

### Taste rubric

Mode: **diagnostic / frontier refinement**  
- Mechanistic grounding: 4 — PR #7000 is merged upstream; the mechanism (CPU round-trip elimination) is direct and well-understood; the only uncertainty is whether it's already enabled in our stack.  
- Research-state value: 4 — If already enabled (no-op), that definitively closes a lane. If not enabled, cheap flag change.  
- Execution value: 4 — One-line config change; zero model change; greedy-exact unaffected; can be confirmed with a 10-minute run.

**This should be a top-priority diagnostic — verify whether it's already on before designing anything else.**

## IDEA 2 — Hydra Sequential MTP Draft Heads (replacing independent Medusa-style heads)

### What it is

Hydra (arxiv 2402.05109, MIT COLM 2024) uses sequentially-dependent draft heads: head k+1 sees the hidden state *conditioned on* the predicted token from head k, rather than all heads being conditioned on the same base hidden state independently (Medusa). This increases acceptance length at the cost of slightly higher per-head compute, netting 1.31x over Medusa and 2.70x over autoregressive baseline.

### Why it might help here

Our current MTP uses K=7 heads with M=8 linear-chain verify, where the 7 draft positions are generated autoregressively by the drafter. Hydra's sequential conditioning is the *correct* approach for maximizing the joint acceptance probability P(t1, t2, ..., tk) because each head can condition on what the previous head predicted. The current independent-head Medusa-style approach — if any — would underestimate E[T] for a given K.

The key question: does our current MTP architecture use sequential conditioning (Hydra-style) or independent conditioning (Medusa-style)? If it uses independent heads, switching to Hydra-style conditioning is a direct E[T] improvement with no token-identity risk (the verify step remains greedy-exact; we just change how drafts are proposed).

### Expected magnitude

Hydra reports 1.31x over Medusa. If our drafter currently uses any form of independent head conditioning, switching to sequential would recover some of that gap. Even a partial gain of +5–10% E[T] improvement (e.g., from E[T]=4.6 to E[T]=5.1) could add +20–40 TPS.

### Key papers

- Hydra (arxiv 2402.05109, 2024): "Sequentially-Dependent Draft Heads for Medusa Decoding" — 1.31x over Medusa; https://arxiv.org/abs/2402.05109
- Medusa (arxiv 2401.10774): Original independent-head approach for comparison baseline.
- EAGLE-2 (arxiv 2406.16858): Adaptive tree construction with context-aware dynamic draft tree.

### Implementation guidance

**File to touch:** The MTP drafter architecture — likely in `model_runner.py` or a custom drafter module under the Gemma model definition.

**Key change:** In the drafter's forward pass, head k should receive as input the embedding of the sampled token from head k-1 (or the argmax under greedy), concatenated with or added to the hidden state. Pseudo-code:
```python
hidden = base_hidden_state
drafts = []
for k in range(K):
    logits_k = head_k(hidden)
    token_k = logits_k.argmax(-1)           # greedy draft
    drafts.append(token_k)
    hidden = hidden + token_embedding(token_k)   # sequential conditioning
```

**Critical check:** First, determine whether the current MTP drafter is already sequential. Read the drafter forward pass. If it already does sequential conditioning, this idea is already captured and can be closed.

**Gotcha:** Sequential conditioning adds one embedding lookup per draft step but is negligibly more expensive than independent heads (embedding table is tiny relative to GEMM cost). The verify step is unchanged — still greedy-exact check against the base model.

**Variant worth trying first:** "Soft" sequential conditioning — use the *probability-weighted* embedding rather than the argmax embedding for head k's hidden state. This keeps the draft generation differentiable for future fine-tuning.

## IDEA 3 — MBQ: Modality-Balanced Quantization for Multimodal Verify Path

### What it is

MBQ (Modality-Balanced Quantization, arxiv 2412.xxxxx, Dec 2024) observes that in VLMs, different modalities have different quantization sensitivity. Text tokens are more robust to weight quantization; visual/audio tokens are more sensitive. Naively applying uniform W4A16 treats all tokens equally and destroys accuracy on visual tasks. MBQ calibrates quantization separately per modality, allowing the text-decode path (our hot path) to use more aggressive quantization while protecting modality-sensitive paths.

### Why it might help here

Gemma-4-E4B-it is a *full* multimodal model — we cannot disable image/audio/video pathways (hard gate). But our benchmark is text-only prompts at inference. MBQ suggests that on the text path we might be able to push quantization more aggressively (e.g., W3 for some layers, or lower-bit scales for the verify-GEMM layers) while the visual pathway remains at W4 — without perplexity regression on text because text quantization sensitivity is lower.

This is a *greedy-lossless* operation: the model weights change, but if PPL stays below the 2.42 gate and greedy tokens match, the change is valid. MBQ specifically targets this PPL-preserving objective.

### Expected magnitude

W3 for the text-critical layers (FFN, self-attn projections) while keeping W4 for visual adapters could reduce memory bandwidth by ~15–20% on those layers, translating to +5–15% TPS on the BW-bound verify GEMM (53% of budget). Rough estimate: +15–25 TPS (+3–5%).

### Key papers

- MBQ (arxiv 2412.xxxxx): "MBQ: Modality-Balanced Quantization for Large Vision-Language Models" — Dec 2024; targeted calibration per modality pathway.
- Cherry on Top (NeurIPS 2024): Small subset of "cherry" parameters disproportionately affect behavior; guides per-layer bit assignment — complementary to MBQ for identifying which layers to quantize more aggressively.
- AMQ (POSTECH 2025): AutoML for mixed-precision weight-only quantization; automatic layer-by-layer bit-width search under PPL budget.

### Implementation guidance

**File to touch:** The quantization configuration / calibration script — likely `quantize.py` or wherever the W4A16 GPTQ/AWQ calibration is run.

**Approach:**
1. Run Cherry on Top's per-layer sensitivity analysis: measure PPL impact of quantizing each layer independently.
2. Identify layers where W4→W3 causes <0.01 PPL increase on text-only calibration data.
3. Create a mixed W4/W3 quantization config (GPTQ supports this via `bits_per_layer` dict).
4. Validate greedy token identity on the 128-prompt benchmark.

**Gotcha:** The PPL gate is tight (~2.42). Any aggressive quantization must be validated on the *exact* 128-prompt greedy decode, not just WikiText-2 PPL. The greedy-exact check is the binding constraint.

**Variant worth trying first:** Target only the FFN down-projection layers (largest W, most BW-bound) for W3, keep all attention projections at W4. FFN down-proj is the most BW-dominant sub-block and typically most robust.

**MBQ reference implementation:** Search GitHub for `mbq-vlm` or check the paper's supplementary for calibration split details.

## IDEA 4 — SWIFT / CAS-Spec: Self-Speculative Decoding via Layer Skipping (No Drafter Model Needed)

### What it is

SWIFT (Self-speculative decoding via feature-based approximate drafting, ICLR 2025, arxiv 2408.11GOP) generates draft tokens by running only a *subset* of transformer layers (skipping the middle layers), then verifying with the full model. No separate drafter model required — the base model itself serves as its own drafter by skipping layers. Achieves 98–100% greedy acceptance rate (because it's running the same weights, just fewer layers) and 1.3–1.6x speedup.

CAS-Spec (NeurIPS 2025) extends this with *cascade adaptive* layer selection — dynamically choosing which layers to skip per token based on confidence, instead of a fixed skip schedule.

### Why it might help here

Our current MTP approach uses a separate drafter model (small Gemma sub-model). Layer-skipping self-speculation eliminates the drafter model entirely, freeing its VRAM and compute budget. On A10G with 24GB and memory-BW-bound single-stream decode:
- No drafter weights loaded = more BW for verify path
- Draft tokens generated by partial forward pass through verify model = no separate DRAM read for drafter weights
- Acceptance rate: 98–100% because draft and verify share all the same weights (skipped layers converge)

The greedy-exact gate is the key concern: SWIFT reports 98–100% acceptance but not necessarily 100%. We need the 100% end — which CAS-Spec achieves by being adaptive (only skips when the model is confident, keeping acceptance near-perfect).

### Expected magnitude

SWIFT: 1.3–1.6x speedup. In our setting, eliminating the drafter (currently ~7% of budget) plus the layer-skip draft savings could push +5–12% TPS. If the layer-skip approach achieves E[T]~3–4 with near-zero draft overhead, the net gain vs. full drafter MTP depends on the E[T] trade-off.

### Key papers

- SWIFT (arxiv 2408.11GOP, ICLR 2025): Self-speculative decoding via layer skipping; 1.3–1.6x; 98–100% acceptance; https://arxiv.org/abs/2408.11GOP  
- CAS-Spec (NeurIPS 2025): Cascade adaptive self-speculative; dynamic layer selection per token confidence.
- DEL (COLM 2025): Dynamic exit layer selection for self-spec decoding; selects optimal exit layer per step.
- EAGLE-2 (2406.16858): Comparison point — EAGLE uses a different drafter architecture; SWIFT/CAS-Spec eliminate drafter entirely.

### Implementation guidance

**Approach:** In Gemma-4's forward pass, add a flag `--draft_skip_layers=[list]` that causes the transformer to skip those layers during draft generation. The verify pass always runs all layers.

**Draft generation:**
```python
def draft_forward(input_ids, skip_layers={8,9,10,11,12,13,14,15}):
    # Run layers 0-7, skip 8-15, continue from 16 onward
    hidden = embed(input_ids)
    for i, layer in enumerate(transformer_layers):
        if i in skip_layers:
            continue    # skip middle layers for draft
        hidden = layer(hidden)
    return lm_head(hidden)
```

**Critical:** Must achieve 100% greedy acceptance for the hard gate. Test with a conservative skip (2–4 middle layers) first. Measure acceptance rate vs. number of layers skipped.

**Gotcha:** Gemma-4 has 46 transformer layers (E4B). Skipping the middle third (~layers 15–30) is typically where SWIFT gets the best speed/acceptance trade-off. Skipping fewer layers → higher acceptance; skipping more → faster draft but lower acceptance.

**Variant:** DEL's approach — binary search per decode step for the deepest layer that still produces a greedy-identical token. This maximizes speed per step while maintaining 100% acceptance.

**File to touch:** `modeling_gemma4.py` or equivalent Gemma-4 forward pass; add `skip_layer_mask` parameter.

## IDEA 5 — LUT-Based W4A16 GEMM (Hardware-Aligned Dequant for A10G Ampere)

### What it is

LUT Tensor Core (arxiv 2408.06003 v2 May 2025) replaces the standard multiply-then-accumulate W4A16 decode path with a lookup-table-based approach: precompute a table of partial products for each weight group, then use tensor-core-friendly table lookups during matmul. On Ampere (A10G = sm_86), this enables using INT8 Tensor Cores for 4-bit weight × 16-bit activation GEMMs via table symmetrization and operator fusion, bypassing the dequant-then-FP16-mul pipeline.

GANQ (ICML 2025) uses a similar LUT-based approach for W4A16 but with GPU-adaptive non-uniform quantization — quantizing weight values to non-uniformly spaced codes that minimize reconstruction error while still enabling fast LUT evaluation.

### Why it might help here

The verify GEMM is 53% of our decode budget, and it is weight-BW-bound at M≤32 with the flat Marlin kernel (W4A16 Marlin). The Marlin SplitK path (ubel #84) attacks the GEMM compute cost; LUT Tensor Core attacks the *dequantization overhead* within the GEMM itself by folding dequant into the TC pipeline.

On A10G (sm_86), INT8 Tensor Cores run at 2x the throughput of FP16 Tensor Cores. If we can express the W4A16 decode GEMM as INT8 table lookups, we potentially double the effective FLOP/s available for the verify GEMM, reducing its wall-clock time by ~30–40%.

The greedy-exact gate: LUT quantization uses the same weight values (just evaluated via table), so if the table is constructed to be numerically equivalent to dequant+multiply (within FP16 rounding), token identity is preserved.

### Expected magnitude

LUT Tensor Core paper reports 1.3–2.1x GEMM speedup on Ampere. Applied to the 53% verify-GEMM budget: (0.53 × 0.4 improvement) / 1.0 = ~21% total step speedup = +100 TPS equivalent. More conservatively, if we get 1.4x on the GEMM, we get +22% on that budget slot, translating to +12% total TPS (~58 TPS). This would clear 500 TPS with substantial margin.

### Key papers

- LUT Tensor Core (arxiv 2408.06003 v2, May 2025): LUT-based W4A16 GEMM for Ampere sm_86/sm_80; operator fusion, table symmetrization; targeted at decode M≤32 regime.
- GANQ (ICML 2025): GPU-adaptive non-uniform quantization with LUT-based evaluation; optimized for A100/A10G; lower quantization error than uniform 4-bit.
- Q-Palette (NeurIPS 2025): Fractional-bit quantizers; ~36% faster decoding; CUDA kernels available.
- LiquidGEMM (arxiv 2509.01229): W4A8 hardware-efficient GEMM — note this requires A8 activation quantization (potentially lossy for greedy), so use with caution.

### Implementation guidance

**Approach:** Swap the Marlin W4A16 verify-GEMM kernel for a LUT-based alternative.

**File to touch:** The GEMM dispatch in `vllm/_core_ext.so` or the custom kernel at `csrc/quantization/marlin/`. Look for where `ops.marlin_gemm` or `marlin_24` is called during the verify forward pass.

**Integration path:**
1. Implement or adopt the LUT Tensor Core kernel (paper provides pseudocode; GANQ has open-source CUDA code at GitHub `IST-DASLab/GANQ`).
2. Replace `marlin_gemm(qweight, scales, input)` calls with `lut_gemm(qweight_lut, input)` in the verify path only (drafter can stay on Marlin).
3. Validate greedy token identity: the LUT must produce bit-identical or within-rounding-tolerance results vs. the current Marlin kernel.

**Gotcha:** Table construction must happen at model load time (offline). The LUT needs to encode the dequantization mapping for each weight group. If GANQ uses non-uniform codes, you need to first re-quantize the checkpoint to GANQ's code format — this is a one-time offline step.

**Critical check:** Does the verify path use Marlin's INT4 weight format or GPTQ's INT4 format? The LUT construction differs. GANQ operates on GPTQ-format weights directly.

**Variant worth trying first:** Use GANQ's published kernel as-is (it has a PyPI package or GitHub release) on the Gemma-4 quantized checkpoint. Benchmark only the linear layers in the verify GEMM (the hot M≤32 path).

**Reference code:** https://github.com/IST-DASLab/GANQ — check the `kernels/` directory for the LUT-based GEMM implementation.

## IDEA 6 — MaskLLM 2:4 Semi-Structured Sparsity on Verify Path

### What it is

MaskLLM (NeurIPS 2024, NVIDIA) learns optimal 2:4 semi-structured sparsity masks end-to-end via Gumbel-Softmax reparameterization. Unlike magnitude pruning which picks the 2 smallest weights per group of 4, MaskLLM trains the sparse mask jointly with model quality, allowing the mask to respect task-relevant weight patterns. NVIDIA A10G supports hardware-accelerated 2:4 sparse GEMM via `cusparseLt`, providing ~1.3–1.5x GEMM throughput (the sparse matmul executes at 2x FLOP efficiency, with ~35% overhead from metadata/indices).

### Why it might help here

At M≤32 (single-stream decode), the verify GEMM is weight-bandwidth-bound — meaning we move the same weight bytes through DRAM per token whether we use dense or sparse. Hardware 2:4 sparsity (cuSPARSELt) cuts the number of non-zero weight bytes read from DRAM by ~50%, directly halving the memory-BW cost of the verify GEMM. This is the *exact* bottleneck identified in the drafter/verify budget breakdown (53% budget on BW-bound Marlin W4A16).

The greedy-exact gate: we need the sparse model to produce identical greedy tokens. This is achievable by:
1. Using MaskLLM to learn the sparse mask (better accuracy preservation than magnitude pruning).
2. Validating greedy token identity on the 128-prompt benchmark before submitting.
3. PPL must remain ≤ 2.42.

### Expected magnitude

2:4 sparsity applied to the W4A16 verify path: ~50% reduction in weight BW → roughly equivalent to going from W4 to W2 in bandwidth terms. At M≤32 (BW-bound regime), the GEMM wall-clock time should roughly halve, but overhead from cusparseLt metadata and activation-side access means a practical speedup of 1.3–1.5x on the GEMM. Applied to 53% of the decode budget: 0.53 × 0.35 = +18.5% total TPS = +89 TPS. This comfortably clears 500 TPS if PPL holds.

**Critical uncertainty:** Does 2:4 sparsity on top of W4A16 quantization preserve greedy token identity? W4A16 is already lossy; adding 50% weight sparsity on top compounds the error. MaskLLM improves mask quality but doesn't guarantee greedy identity — PPL-equivalence is the practical gate here.

### Key papers

- MaskLLM (NeurIPS 2024, NVIDIA, arxiv 2409.17481): End-to-end learnable 2:4 sparse mask training; outperforms magnitude pruning; A10G-compatible via cuSPARSELt.
- V:N:M Sparsity (ICLR 2025 submission): Channel permutation to achieve hardware-accelerated V:N:M patterns beyond standard 2:4; more flexible sparsity ratios.
- SparseGPT (arxiv 2301.00774, 2023): One-shot unstructured pruning for LLMs; comparison baseline for structured sparsity.
- Wanda (arxiv 2306.11695, 2023): Weight pruning using activation norms — simpler than MaskLLM but less accurate mask.

### Implementation guidance

**Approach:**
1. Take the existing W4A16 quantized Gemma-4-E4B-it checkpoint.
2. Run MaskLLM mask training on top: learn 2:4 masks for the verify-path linear layers (FFN + attention projections in transformer layers 0–45).
3. Apply cuSPARSELt-format compression to the W4A16 weights (requires converting to sparse format compatible with cuSPARSELt's expected layout).
4. Validate PPL and greedy token identity.

**File to touch:** Model loading / checkpoint conversion — likely `convert_checkpoint.py` or equivalent. The cusparseLt kernel call would go where `marlin_gemm` is currently called.

**Gotcha 1:** cuSPARSELt requires weights in a specific compressed format (not compatible with Marlin's INT4 layout directly). You need to either:
- Use W16 (FP16) sparse weights with cuSPARSELt (loses quantization gain), OR
- Implement a custom kernel that combines W4 dequant + 2:4 sparse matmul (what LUT Tensor Core above does implicitly).

**Gotcha 2:** MaskLLM requires a calibration dataset and a fine-tuning step (~100–1000 gradient steps). This is more expensive than one-shot magnitude pruning. Use C4 or the model's original training distribution for calibration.

**Variant worth trying first:** One-shot Wanda-pruned 2:4 sparsity (no training required) as a quick PPL sanity check. If PPL stays ≤ 2.42 with simple Wanda, MaskLLM will do strictly better. Wanda takes ~30 min on a single GPU.

**Reference code:** https://github.com/NVlabs/MaskLLM — includes training scripts and cuSPARSELt integration examples.

## IDEA 7 — Blockwise Parallel Decoding + Draft Refinement (Lossless Multi-Draft)

### What it is

Blockwise Parallel Decoding (arxiv 1811.03115, Google Brain, 2018 — a Schmidhuber-style old idea) proposes training auxiliary prediction heads at every layer to predict tokens N steps ahead, then verifying with the base model. The key insight: predicting block (t, t+1, ..., t+k) in parallel at one forward pass, then accepting the longest greedy-matching prefix. This is the conceptual ancestor of Medusa/EAGLE.

The modern revival (NeurIPS 2024 "Blockwise Parallel + Draft Refinement") adds a *draft refinement* step: after generating the initial block draft, run a cheap "refinement" pass that corrects the most obvious draft errors before verification, improving the block acceptance rate without extra base-model cost.

### Why it might help here

Our current MTP with K=7 heads operates linearly: draft head 1 predicts t+1, head 2 predicts t+2, ..., head 7 predicts t+7. The accept length E[T] ≈ 4.6–5.2. Draft refinement is orthogonal to the number of heads: it improves the *quality* of the K-step draft, potentially increasing E[T] from 4.6 to 5.5–6.5 without changing K or the verify overhead.

The mechanism: after generating draft tokens with the current MTP drafter, run a single cheap correction pass (a 1–2 layer transformer on the draft sequence) that uses the context of all K draft tokens simultaneously to "self-correct" obvious inconsistencies. This refinement is cheap (1–2 FFN layers) but can catch systematic draft errors.

### Expected magnitude

NeurIPS 2024 paper reports +15–25% improvement in block acceptance rate with draft refinement. If our current E[T]=4.6 and refinement improves it to E[T]=5.3, that's +15% in TPS from the accept-length lever alone = +72 TPS → 553 TPS. More conservatively, +10% E[T] improvement = +48 TPS → 530 TPS. Still clears 500 TPS comfortably.

### Key papers

- Blockwise Parallel Decoding (arxiv 1811.03115, 2018): Original proposal; trains aux heads at each layer; conceptual ancestor of Medusa. https://arxiv.org/abs/1811.03115
- Draft Refinement (NeurIPS 2024, "Blockwise Parallel Decoding with Draft Refinement Algorithms"): Adds a cheap self-correction pass on the draft sequence before verification. 
- Lookahead Decoding (arxiv 2402.02057, ICML 2024): N-gram tree + verification — related but different mechanism (lookup vs. generation).

### Implementation guidance

**File to touch:** The drafter module — wherever the K=7 MTP draft tokens are generated before verification.

**Draft refinement architecture:**
```python
class DraftRefiner(nn.Module):
    """Lightweight 2-layer transformer to self-correct the K-step draft."""
    def __init__(self, d_model, K):
        self.layers = nn.ModuleList([TransformerLayer(d_model) for _ in range(2)])
        self.proj = nn.Linear(d_model, vocab_size)
    
    def forward(self, draft_tokens, draft_hidden_states):
        # draft_hidden_states: [B, K, d_model] from the drafter
        x = draft_hidden_states
        for layer in self.layers:
            x = layer(x)
        refined_logits = self.proj(x)    # [B, K, vocab]
        return refined_logits.argmax(-1)  # refined draft tokens
```

**Training:** Requires a quick fine-tuning step to train the refiner on (draft_tokens, correct_tokens) pairs from the base model. ~10K steps on the calibration corpus.

**Gotcha:** The refiner adds ~5–10ms per decode step (2 FFN layers over K=7 tokens). Must verify that the latency savings from higher E[T] outweigh the refiner's overhead. Break-even: if refiner adds 8ms per step and E[T] improves from 4.6 to 5.3, net gain is (5.3-4.6)×step_time - 8ms. Needs profiling.

**Greedy-exact gate:** The refiner changes the *draft* tokens, not the verification. The base model still does greedy verification — only accepts tokens where verify logit argmax == draft token. Token identity is preserved.

## IDEA 8 — KVTuner Layer-Wise Mixed-Precision KV Cache Quantization

### What it is

KVTuner (ICML 2025) performs sensitivity-aware automatic search for per-layer KV cache quantization precision — assigning KV8 to sensitive layers and KV4 to robust ones, minimizing PPL degradation under a target memory budget. Unlike uniform KV4 (which degrades accuracy) or uniform KV8 (which wastes memory), KVTuner finds the Pareto-optimal mixed-precision assignment.

### Why it might help here

At conc=1 single-stream decode with output_len=512 and 128 prompts, the KV cache for a single sequence fits comfortably in A10G's 24GB. KV cache quantization in this regime doesn't help VRAM — but it does help *memory bandwidth*: loading KV8 vs. KV16 for attention computation halves the BW cost of attention reads. At K=7 MTP, the verify attention reads K+1 KV entries per step. KV cache BW is a sub-dominant term (<8% of budget), but at the frontier where we need +3.8%, even sub-dominant improvements compound.

More importantly: KVTuner's per-layer mixed precision creates headroom to compress KV aggressively on robust layers while maintaining accuracy on sensitive ones, potentially enabling effective KV4 on most layers — a ~2x reduction in attention BW.

### Expected magnitude

KV attention is <8% of current decode budget. Even 2x reduction in KV BW = +4% on that component = +0.32% total TPS = ~1.5 TPS. This alone doesn't clear 500 TPS. However, combined with other improvements it contributes, and the real value is if KV compression frees DRAM bandwidth for the verify GEMM (shared HBM controller).

**Assessment:** This is a lower-priority idea that helps at the margin. Worth trying after higher-impact ideas (Ideas 1–3) are exhausted.

### Key papers

- KVTuner (ICML 2025): Per-layer mixed-precision KV quantization; sensitivity-aware search.
- KVQuant (NeurIPS 2024): Per-channel Key quantization, Pre-RoPE Key quantization for sub-4-bit KV.
- MiniCache (NeurIPS 2024): KV cache compression in the depth (layer) dimension; near-lossless cross-layer KV merging.
- Ecco (arxiv 2505.06901): Entropy-aware KV compression — skip KV storage for tokens with high prediction certainty.

### Implementation guidance

**File to touch:** KV cache allocation in vLLM — `vllm/attention/backends/flash_attn.py` or the KV cache manager.
